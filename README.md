# Provenance Guard

## Overview

Provenance Guard is a Flask-based backend application designed to help estimate whether submitted text is likely written by a human or generated by artificial intelligence. Rather than relying on a single detection method, the system combines multiple independent detection signals into a single confidence score and provides a transparent explanation of how that decision was reached.

The project emphasizes responsible AI detection by including confidence scoring, transparency labels, audit logging, an appeals workflow, rate limiting, and input validation. Instead of making absolute claims about authorship, the system communicates uncertainty and allows creators to challenge incorrect classifications through a built-in appeals process.

---

# Features

- Multi-signal AI detection
- LLM-based AI detection using the Groq API
- Stylometric writing analysis
- Confidence score calculation
- Three transparency label categories
- Audit logging for all submissions
- Appeals workflow for disputed classifications
- Rate limiting using Flask-Limiter
- JSON input validation
- RESTful API endpoints

# Detection Signals

Provenance Guard uses two independent detection signals to estimate whether submitted text was written by a human or generated by an AI model. Rather than relying on only one detector, the system combines multiple signals to produce a more balanced result.

## Signal 1 – LLM Detector

### What it measures

The first detector uses the Groq API with the Llama 3.3-70B model to evaluate how likely a piece of text is AI-generated.

The model examines characteristics such as:

- Overall writing consistency
- Sentence structure
- Predictability
- Fluency
- AI-like writing patterns

### Output

The detector returns a score between **0 and 100**.

| Score | Meaning |
|--------|---------|
| 0 | Definitely Human |
| 100 | Definitely AI |

### Why this signal was chosen

Large language models have learned many characteristics of AI-generated writing and can recognize common stylistic patterns.

### What it misses

The detector may incorrectly classify:

- Highly polished human writing
- Academic essays
- Technical documentation
- Short pieces of text with limited context

---

## Signal 2 – Stylometric Analysis

### What it measures

The stylometry detector analyzes characteristics of the writing itself rather than asking another language model.

Current heuristics examine features such as:

- Sentence length
- Word usage
- Writing complexity
- Overall writing style

### Output

The stylometry detector also returns a score between **0 and 100**.

Example

```
Stylometry Score = 50
```

### Why this signal was chosen

Stylometric analysis provides an independent source of evidence that does not depend on another language model.

### What it misses

Simple heuristics cannot capture every writing style.

For example, they may struggle with:

- Poetry
- Song lyrics
- Creative writing
- Very short responses

---

# Confidence Scoring

Instead of allowing either detector to make the final decision alone, Provenance Guard combines both scores into a single confidence score.

Current implementation:

```
Confidence = (LLM Score + Stylometry Score) / 2
```

Example

```
LLM Score = 80
Stylometry Score = 50

Confidence = 65
```

The confidence score is then mapped to one of three classifications.

| Confidence Score | Final Label |
|-----------------|-------------|
| 0–39 | Likely Human |
| 40–69 | Uncertain |
| 70–100 | Likely AI |

The "Uncertain" category is intentionally wide because false accusations are more harmful than occasionally missing AI-generated content. When the detectors disagree or produce only moderate evidence, the system avoids making a strong claim.

# Audit Logging

Every successful submission is recorded in an audit log stored in `storage/audit_log.json`. Logging each request improves transparency and provides a record that can be reviewed later if a user disputes a classification.

Each audit log entry contains:

- Creator ID
- Submitted text
- LLM detector score
- Stylometry score
- Confidence score
- Final classification
- Timestamp

Example audit log entry:

```json
{
    "creator_id": "yakubu",
    "text": "This is another audit log test.",
    "llm_score": 90,
    "style_score": 50,
    "confidence": 70,
    "label": "Likely AI",
    "timestamp": "2026-06-30T23:17:32.939683"
}
```

Audit logs can be viewed through the `GET /logs` endpoint.

# Appeals Workflow

Because AI detection is imperfect, Provenance Guard includes an appeals process that allows creators to request manual review of a classification.

### Submission

A creator submits:

- Creator ID
- Written explanation describing why they believe the classification is incorrect

Example:

```json
{
    "creator_id":"yakubu",
    "reason":"I believe my submission was incorrectly classified as AI-generated."
}
```

---

### Processing

When an appeal is received:

- The appeal is saved in `storage/appeals.json`
- A timestamp is recorded
- The appeal status is initialized as **Pending**

Example:

```json
{
    "creator_id":"yakubu",
    "reason":"I believe my submission was incorrectly classified.",
    "status":"Pending",
    "timestamp":"2026-06-30T23:34:47.231779"
}
```

---

### Human Review

A reviewer can view:

- Creator ID
- Appeal reason
- Timestamp
- Current appeal status

Future versions of the system may allow reviewers to update appeals from **Pending** to **Approved** or **Rejected**.

# Rate Limiting

The `/submit` endpoint is protected using Flask-Limiter.

Current configuration:

```
5 requests per minute
```

### Why 5 per minute?

The goal is to discourage automated abuse while allowing normal users to submit multiple drafts during editing.

A writing platform typically does not require dozens of submissions every minute. Limiting requests reduces opportunities for attackers to flood the detection service or repeatedly probe the detector for weaknesses.

---

# Known Limitations

Although Provenance Guard combines multiple detection signals, it cannot perfectly determine authorship.

### Limitation 1

A poem containing repetitive language, intentionally simple vocabulary, or repeated sentence structures may resemble AI-generated writing even though it was written by a human.

This affects both the LLM detector and the stylometric analysis.

---

### Limitation 2

Highly polished academic writing or technical documentation may receive elevated AI scores because the writing is unusually consistent and grammatically correct.

The LLM detector is more likely to misclassify this type of content.

---

### Limitation 3

Very short submissions provide limited evidence for both detectors, making confidence scores less reliable.

For this reason, users should interpret low-confidence classifications carefully.

# Spec Reflection

The original project plan proposed combining multiple detection signals into a single confidence score while providing transparent explanations for every decision.

During implementation, one notable change was simplifying the confidence calculation. Rather than designing a more complex weighting system, the final implementation averages the LLM detector score and the stylometric score. This approach made the system easier to explain, test, and debug while still demonstrating how multiple signals can be combined.

Another improvement beyond the original plan was adding input validation to ensure that required fields such as `text` and `creator_id` are present before processing a submission. This made the API more robust and user-friendly.

# Future Improvements

Several enhancements could improve Provenance Guard in future versions.

- Replace simple stylometric heuristics with richer linguistic features.
- Introduce a weighted ensemble using three or more detection signals instead of averaging two scores.
- Add reviewer tools for managing appeals directly through the API.
- Replace JSON storage with a relational database for scalability.
- Add user authentication and verified creator identities.
- Build a web dashboard to visualize detection statistics and appeal outcomes.
- Extend the system to support additional content types such as image captions or structured metadata.

# Conclusion

Provenance Guard demonstrates a practical approach to AI content detection by combining multiple detection signals with confidence scoring, transparent explanations, audit logging, and a human appeals process. Rather than presenting AI detection as certain, the system communicates uncertainty honestly and provides users with mechanisms to review and challenge automated decisions.
