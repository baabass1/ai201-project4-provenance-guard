# Provenance Guard - Planning Document

## Project Overview

Provenance Guard is a Flask-based backend that helps identify whether submitted text is likely AI-generated or human-written. Rather than relying on a single detector, the system combines multiple detection signals, calculates an overall confidence score, provides a transparent explanation of the decision, stores an audit log of every submission, supports user appeals, and limits excessive API requests.

---

# 1. Detection Signals

The system uses two independent detection signals.

## Signal 1: LLM-Based Detector

**Purpose**

The first signal uses a large language model (Groq Llama 3.3-70B) to estimate how likely a piece of text was generated by AI.

**Measures**

- Writing consistency
- Sentence structure
- Predictability
- Overall AI-like writing style

**Output**

An integer score from **0–100**.

- 0 = Definitely Human
- 100 = Definitely AI

Example:

```
LLM Score: 82
```

---

## Signal 2: Stylometric Analysis

**Purpose**

The second signal analyzes writing style using simple stylometric heuristics.

Current implementation considers characteristics such as:

- Sentence length
- Word patterns
- Overall writing complexity

This detector serves as an independent signal rather than relying on the language model.

**Output**

An integer score from **0–100**.

Example:

```
Stylometry Score: 50
```

---

## Combining Signals

The system combines both signals by averaging their scores.

Example:

```
LLM Score = 80
Stylometry Score = 50

Confidence = (80 + 50) / 2 = 65
```

The combined confidence score is then used to determine the final classification.

---

# 2. Uncertainty Representation

The confidence score ranges from **0–100**.

The system uses three confidence regions.

| Confidence Score | Classification |
|-----------------|---------------|
| 0–39 | Likely Human |
| 40–69 | Uncertain |
| 70–100 | Likely AI |

### Example

A confidence score of **60** means:

- The available evidence is mixed.
- The detectors do not strongly agree.
- The system cannot confidently classify the submission as AI-generated or human-written.

Instead of making a strong claim, the system returns **Uncertain**.

This reduces the likelihood of overconfident false positives.

# 3. Transparency Label Design

Every submission returns a transparency object explaining how the decision was made.

## High Confidence AI

```
Final Label:
Likely AI

Confidence:
85/100

Explanation:
The AI detector and stylometric analysis both indicate strong evidence that this text was AI-generated.
```

---

## High Confidence Human

```
Final Label:
Likely Human

Confidence:
25/100

Explanation:
Both detection methods suggest that the writing is consistent with human-written text.
```

---

## Uncertain

```
Final Label:
Uncertain

Confidence:
55/100

Explanation:
The available signals disagree or provide only moderate evidence. Human review may be appropriate.
```

# 4. Appeals Workflow

Any content creator may submit an appeal if they believe their submission was incorrectly classified.

### Information Required

- Creator ID
- Written explanation describing why the result should be reviewed

### System Behavior

When an appeal is submitted:

- The appeal is stored in `storage/appeals.json`
- A timestamp is recorded
- The appeal status is set to **Pending**

### Human Reviewer

When reviewing appeals, a reviewer can see:

- Creator ID
- Appeal reason
- Timestamp
- Current appeal status

This information provides enough context for manual review while keeping the workflow simple.

### Appeal Queue

Each new appeal enters the review queue with a status of **Pending**. After manual review, the appeal may be updated to either **Approved** or **Rejected**.

# 5. Anticipated Edge Cases

## Edge Case 1

A poem containing repetitive language, short sentences, or intentionally simple vocabulary may resemble AI-generated writing even though it was written by a person.

---

## Edge Case 2

A human writer using grammar correction software or heavily editing their work may produce writing that appears unusually polished, increasing the AI score.

---

## Edge Case 3

Very short submissions (for example only one or two sentences) provide limited evidence and may produce unreliable confidence scores.

# AI Tool Plan

AI tools were incorporated throughout both the planning and implementation phases of this project to accelerate development, troubleshoot technical issues, and improve documentation. AI assistance was used as a software development aid rather than a replacement for engineering judgment. Every AI-generated suggestion was reviewed, modified, tested, and integrated manually before becoming part of the final system.

## Planned AI-Assisted Tasks

### 1. Backend API Development

AI assisted in designing the initial Flask backend structure, including:

- Flask application organization
- REST API endpoint templates
- Blueprint structure
- Request and response formatting
- JSON response generation

Inputs provided to AI included:

- The project specification and grading rubric
- Required API endpoints
- Expected request and response formats
- Planned backend architecture

The generated code served only as a starting point and was later modified to include confidence scoring, transparency labels, audit logging, appeals, input validation, and rate limiting.

---

### 2. Detection Pipeline Design

AI assisted in brainstorming how multiple detection signals could be combined into a single confidence score.

Inputs provided included:

- Detection signal definitions
- Confidence score requirements
- Classification thresholds
- Transparency label requirements
- Planned system architecture diagram

The final implementation combines the outputs of an LLM-based detector and a stylometric detector by averaging their scores before assigning one of three confidence labels: **Likely Human**, **Uncertain**, or **Likely AI**.

---

### 3. Debugging and Refactoring

AI was used throughout development to diagnose implementation issues and suggest improvements to the project structure.

Examples include:

- Resolving circular import issues involving Flask blueprints
- Organizing the application into `routes`, `detectors`, `services`, `storage`, and `utils`
- Integrating Flask-Limiter for rate limiting
- Improving input validation
- Refactoring code into reusable modules

Each suggested solution was manually reviewed and tested before being incorporated into the project.

---

### 4. Documentation Support

AI assisted in organizing and drafting project documentation.

This included assistance with:

- Planning document organization
- README structure
- API documentation
- Architecture descriptions
- Detection pipeline explanation
- Transparency label descriptions
- Appeals workflow documentation

Inputs provided included:

- The completed implementation
- Project specification and rubric
- Detection workflow
- Planned system architecture diagram
- Confidence scoring strategy

The final documentation was manually edited to ensure it accurately reflected the implemented system rather than the initial AI-generated drafts.

---

## Human Oversight

Although AI significantly accelerated development, all major design decisions—including the system architecture, detection workflow, confidence scoring strategy, transparency labels, audit logging, appeals workflow, rate limiting configuration, and documentation—were reviewed, modified, tested, and validated manually before submission. AI served as a development assistant rather than an autonomous software engineer.
